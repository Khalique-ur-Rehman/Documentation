1. Introduction

1.1 Purpose

This document describes four custom Odoo 18 modules used together to implement a complete University Management and Outcome‑Based Education (OBE) solution:

  • UMS Module – manages core university data such as dashboards, courses, people, and departments.
  • Custom Quest Module – manages the full lifecycle of exam papers (building, submission, moderation, and scheduling).
  • Activity Logger Module – records detailed user activity (sessions, requests, database changes) for audit and troubleshooting.
  • OBE System Module – manages OBE structures (documents, PEOs, PLOs, CLOs) and links them to courses and assessments.

1.2 Scope

The guide explains:

  • Main screens provided by each module.
  • Navigation paths.
  • Key fields and their meaning.
  • Typical workflows and how the four modules connect.

Low‑level technical topics (source code, deployment, and security configuration) are not covered.

1.3 Target Audience

  • System administrators and examination controllers.
  • Heads of Department (HODs).
  • Faculty members and course instructors.
  • Academic and HR office staff.
  • Quality assurance and accreditation teams.
  • Auditors who review logs and OBE structures.

2. UMS Module

2.1 Overview

The UMS module is the central point for managing university‑wide information. It provides:

  • Dashboards:
    • UMS Dashboard (university‑level view).
    • HOD Dashboard (department‑level view).
  • Academic structure:
    • Courses and their prerequisites/co‑requisites.
    • Departments and their hierarchy.
  • People:
    • Students, Staff, Faculty, and a combined All Users view.

2.2 UMS Dashboard

Navigation: UMS System → Dashboard (UMS Dashboard)

Figure 1: UMS Dashboard – university‑wide statistics

The UMS Dashboard shows high‑level statistics for the entire university.

Key cards:

  • Students
    • Total number of students.
    • Breakdown by status: Active, Graduated, Dropped.
  • Faculty Members – total teaching staff across all departments.
  • Staff Members – total non‑teaching staff.
  • Departments – total departments with counts of Teaching vs Non‑teaching departments.

Usage:

  • Management gets a quick overview of student population, human resources, and departments.
  • If enabled, clicking a card opens the underlying list view (for example, the list of students).

2.3 Courses

Navigation: UMS System → Courses

Figure 2: Courses – list view

Purpose:

Maintain the list of courses offered by the university, including basic details and prerequisite structure.

Main columns:

  • Course Code – e.g., CS101.
  • Course Title – e.g., “Database Management Systems”.
  • Prerequisites – courses that must be completed before this course.
  • Co‑requisites – courses that must be taken alongside this course.

Typical actions:

  • Create a new course
    1. Click “Create”.
    2. Enter Course Code and Course Title.
    3. Add credits, level/semester, department, etc.
    4. Choose prerequisites and co‑requisites if applicable.
    5. Save.
  • Edit existing courses to update titles, codes, or relationships.
  • Use search and filters to find courses by code, title, department, or level.

2.4 HOD Dashboard

Navigation: UMS System → HOD Dashboard

Figure 3: HOD Dashboard – department‑level summary

The HOD Dashboard provides a department‑specific view for the logged‑in Head of Department.

Main cards:

  • Students – number of students in that department.
  • Faculty – faculty members belonging to the department.
  • Courses – courses owned by the department.

Quick Actions section:

  • View Students – opens students filtered to this department.
  • View Faculty – shows only faculty belonging to the department.
  • View Courses – shows the department’s courses.

Usage:

  • HODs quickly see the size and structure of their department.
  • Quick Actions allow direct navigation to relevant lists.

2.5 People

The People area manages all individuals associated with the university. It typically contains four menus: Students, Staff, Faculty, and All Users.

2.5.1 Students

Navigation: UMS System → People → Students

Figure 4: Student – creation form

Purpose:

Store personal, academic, and account information for each student and optionally create linked Odoo user accounts.

Main sections and fields:

A. Basic Info

  • Name – full name.
  • Partner Type – automatically “Student”.
  • Department – department or program.
  • Personal Email – personal contact address.
  • University Email – institutional email.
  • Roles – additional system roles if needed.

B. Student Info

  • Enrollment Number – official enrollment identifier.
  • Roll Number – departmental or batch roll number.
  • Additional fields as configured (semester, session, etc.).

Creating a student:

  1. Click “Create”.
  2. Fill all required fields under Basic Info and Student Info.
  3. If the module supports it, click “Create Student Account” to automatically create an Odoo user for this student.
  4. Save.

Form view – main sections:

  • Partner Type – “Staff”.
  • Department.

  • Roles – functional or system roles.

B. Staff Info

Navigation: UMS System → People → Faculty

Figure 7: Faculty – creation form

2.6 Departments

Figure 9: Departments – list view

Manage academic and administrative departments.

Main columns:

  • Short Name – e.g., “CS”.

3. Custom Quest Module

3.1 Overview

The Custom Quest module manages the complete lifecycle of exam papers inside UMS. It is tightly integrated with UMS data (courses, departments, people) and with CLOs in the OBE System module.

Main components:

  • Controller Dashboard – high‑level status of all papers.
  • Paper Submission – repository of submitted exam papers.
  • Exam Builder – tool for constructing exams (sections, questions, marks, CLOs).
  • Moderator Review – interface for academic review and approval.
  • Academic Calendar – planning and notification of academic events, especially exams.

3.2 Getting Started

3.2.1 Prerequisites

  • Odoo 18 with the Custom Quest module installed.
  • User roles set up for Controller, Instructor, Moderator, and Academic Office staff.

3.2.2 Accessing Custom Quest

  1. Log in to Odoo and open the UMS System.
  2. Click the “Quest” menu in the top navigation bar.
  3. Use the submenu items to open: Controller Dashboard, Paper Submission, Exam Builder, Moderator Review, Academic Calendar.

3.3 Controller Dashboard

Navigation: Quest → Controller Dashboard

Figure 10: Controller Dashboard – paper statistics

Purpose:

Give the Examination Controller a real‑time summary of exam papers and their status.

The dashboard typically shows:

  • Central “Papers” card – total number of papers.
  • Status cards:
    • Under Review – papers currently being reviewed.
    • Revision – papers returned to authors for changes.
    • Accepted – approved papers ready for use.

Usage:

  • Verify that all courses have submitted their required exam papers.
  • Monitor how many papers are still under review or in revision close to exam dates.
  • Ensure that enough papers are accepted before printing or publishing.

3.4 Paper Submission

Navigation: Quest → Paper Submission

Figure 11: Paper Submission – list view

Purpose:

Store and track exam papers submitted by instructors and link them to courses and reviewers.

Typical columns:

  • Paper Title – name of the exam paper.
  • Author Name – instructor who created the paper.
  • Submission Date.
  • Category – exam type (Midterm, Final, Quiz, etc.).
  • Main Reviewer – assigned moderator.
  • Status – current state: Draft, Under Review, Revision, Accepted, etc.

Creating a submission:

  1. Click “Create” or “+ Paper Submission”.
  2. Enter Paper Title and link it to the appropriate Course/Department.
  3. Choose Category and assign a Main Reviewer.
  4. Attach the exam document or link it to the corresponding Exam Builder record.
  5. Save; set Status as Draft or Under Review according to your workflow.

Managing submissions:

  • Use filters to view papers by course, author, reviewer, or status.
  • Update Status as moderators accept or return papers for revision.

3.5 Exam Builder

Navigation: Quest → Exam Builder

Figure 12: Exam Builder – list view

The Exam Builder is used by instructors to design structured exam papers. Each record represents a specific exam instance.

List view columns:

  • Exam Title – e.g., “Final Term Examination”.
  • Course – associated course.
  • Exam Type – Midterm, Final, Quiz, etc.
  • Date – scheduled exam date.
  • Marks – total marks for the exam.
  • Status – Draft, Building, Review, Approved, Rejected.
Figure 13: Exam Builder – exam form (Builder tab)

Exam form – main elements:

  • Buttons:
    • Start Building – move exam into building mode so sections and questions can be added.
    • Preview – open a read‑only preview of the exam layout.
  • Status bar: Draft → Building → Review → Approved → Rejected.
  • Tabs:
    • Exam Configuration – basic info: course, exam type, date, duration, total marks.
    • Exam Header – header text and instructions displayed on the exam paper.
    • Builder – main tab for adding sections and questions.
    • Details – additional metadata or internal notes.

3.5.1 Builder Tab – Structure

Inside the Builder tab you typically have the following sub‑tabs:

  • Questions – table of all questions in the exam.
  • Sections – definitions of exam sections (e.g., Section 01, Section 02).
  • CLO Analysis – mapping of questions to Course Learning Outcomes.

Questions table fields:

  • Section – section name/number.
  • Question Type – Multiple Choice, Short Question, Essay, etc.
  • Question Text – question statement.
  • Has Multiple … – indicates if multiple correct answers are allowed.
  • Marks – marks allocated to the question.
  • CLO – CLO number or code linked to the question.

3.5.2 Adding Sections and Questions

Adding a section:

  1. Open the exam and go to Builder → Sections.
  2. Click “Add a line”.
  3. Enter the section title, order, and any section‑level instructions.
  4. Save.

Adding a question:

  1. Go to Builder → Questions.
  2. Click “Add Question” or “Add a line”.
  3. Select or create the Section.
  4. Choose Question Type.
  5. Enter the Question Text.
  6. Set the Marks and select a CLO.
  7. Enable “Has Multiple …” if there are multiple correct answers.
  8. Repeat for all questions.

3.5.3 CLO Analysis

Purpose:

Ensure that the exam assesses the correct Course Learning Outcomes and that marks are distributed appropriately.

Usage:

  • Open the CLO Analysis sub‑tab to see how many marks are mapped to each CLO.
  • Adjust questions or marks if certain CLOs are under‑ or over‑represented.

3.6 Moderator Review

Navigation: Quest → Moderator Review

Figure 14: Moderator Review – list view

Purpose:

Provide moderators with a list of papers assigned to them for review and capture ratings and decisions.

Typical columns:

  • Paper Title.
  • Author Name.
  • Reviewer.
  • Rating.
  • Status – Under Review, Revision, Accepted, etc.

Review process:

  1. Moderator opens a paper from the Moderator Review list.
  2. Reviews questions, marks distribution, difficulty, and CLO mapping.
  3. Adds comments and sets Rating if required.
  4. Updates Status:
    • Accepted – paper is approved for final use.
    • Revision – paper is returned to the author with feedback for changes.

3.7 Academic Calendar

Navigation: Quest → Academic Calendar

Figure 15: Academic Calendar – list of academic events

Purpose:

Schedule and manage academic events, especially exams, and send reminders.

Columns:

  • Event Name.
  • Event Type – Exam, Workshop, Conference, etc.
  • Start Date.
  • Location.
  • Status – Planned, Confirmed, Completed, Cancelled, etc.

Creating an event:

  1. Click “Create”.
  2. Enter Event Name and Event Type.
  3. Set Start Date and Location.
  4. Fill any additional fields as defined by your system.
  5. Save. For exam events, notifications (for example, 14 days before) are sent to staff.

Managing events:

  • Edit events to change dates or details.
  • Update Status as events are confirmed or completed.
  • Filter by type or date range to view only upcoming exams.

3.8 End‑to‑End Workflow

The following high‑level workflow explains how Custom Quest is used from planning to finalization of exams.

  1. Plan Exams – The Academic Office uses the Academic Calendar to create exam events for each course and semester.
  2. Build Exams – Instructors open Exam Builder and create exams (Midterm, Final, etc.), defining sections, questions, marks, and CLO mappings. Exam status moves from Draft to Building.
  3. Submit Papers – Once complete, the exam paper is linked to a Paper Submission record, assigned a Main Reviewer, and set to Under Review.
  4. Moderate Papers – Moderators review via Moderator Review, rate papers, and set Status to Accepted or Revision.
  5. Revise (if required) – Authors update exams based on feedback and resubmit for another review.
  6. Monitor Progress – The Controller uses the Controller Dashboard to ensure all papers reach Accepted before the scheduled exam dates.
  7. Finalize Exams – Accepted papers are ready for printing and use on exam day.

4. Activity Logger Module

4.1 Overview

The Activity Logger module records detailed user activity in the Odoo system. Its objectives are auditability, traceability, and support for troubleshooting.

Main components:

  • Sessions – user login sessions.
  • Requests – each HTTP/API request made during a session.
  • Changes – database operations triggered by a request.
  • Activity Report – consolidated reporting view.
  • Rules – configuration of what is tracked and at what level of detail.

4.2 Sessions

Navigation: Activity Logger → Sessions

Figure 16: Sessions – form view

Purpose:

Group all requests made by a user during a single login session.

Main fields:

  • Session – unique session identifier (e.g., “SESS/000001”).
  • User – user associated with the session.
  • Create Date – when the session started.

The Requests tab lists all requests belonging to this session.

4.3 Requests

Navigation: Activity Logger → Requests

Figure 17: Requests – form view

Purpose:

Capture each HTTP/API request, including information on user, path, and context.

Main fields:

  • Path – URL path accessed.
  • Root URL – base system URL.
  • Method – HTTP method (GET, POST, PUT, DELETE, etc.).
  • User – who made the request.
  • Session – link to the related Session.
  • Context JSON – raw JSON context (language, timezone, etc.).

The Changes tab lists all data changes triggered by the request.

4.4 Changes

Navigation: Activity Logger → Changes

Figure 18: Changes – form view

Purpose:

Store record‑level operations (create, write, delete) executed on Odoo models.

Main fields:

  • Change – internal reference or name.
  • Model – technical model name affected.
  • Record IDs (csv) – IDs of affected records.
  • Method – type of operation (create, write, unlink, etc.).
  • Mode – Quick or Detailed logging.
  • User – user who performed the operation.
  • Request – related Request.
  • Session – related Session.
  • Create Date – timestamp.

The Field Changes tab contains field‑level details: Field Name, Field Label, Old Value Text, New Value Text.

4.5 Activity Report

Navigation: Activity Logger → Activity Report

Figure 19: Activity Report – consolidated view

Purpose:

Provide a flattened view of change history with key fields for reporting and export.

Columns typically:

  • Create Date.
  • Model.
  • Res Ids Text – text for affected records.
  • Change Method – create, write, unlink, etc.
  • Field Name.
  • Old Value Text.
  • New Value Text.
  • Path – URL path that triggered the change.
  • User – who made the change.

Usage:

  • Filter by Model (e.g., ums.student) to see all changes for that model.
  • Filter by User or date range for audit purposes.
  • Export to Excel/CSV for offline analysis and compliance reports.

4.6 Rules

Navigation: Activity Logger → Rules

Figure 20: Rules – configuration form

Purpose:

Control which models and operations are logged, and how detailed the logging is.

Main fields:

  • Name – descriptive rule name (e.g., “Track Student Records”).
  • Active – whether the rule is enabled.
  • Model – model to monitor (e.g., ums.student, obe.clo).
  • Mode – logging mode:
    • Detailed – include field‑level changes.
    • Quick – record only summary change information.

Operations section:

  • Track Create – log record creation.
  • Track Write – log updates.
  • Track Unlink – log deletions.
  • Track Read – log reads (optional, heavier).
  • Track Export – log exports (optional).

Exclusions section (if configured):

  • Fields or users to exclude from logging.

4.7 Typical Usage and Workflow

  1. Configure logging Rules for important models.
  2. Users operate normally; Activity Logger creates Sessions, Requests, and Changes automatically.
  3. When an issue arises, administrators use Activity Report to filter by model, user, and date.
  4. They drill down from Activity Report → Change → Request → Session to see full context.
  5. Exports from Activity Report can be used in formal audits or accreditation evidence.

5. OBE System Module

5.1 Overview

The OBE System module supports Outcome‑Based Education by structuring program and course outcomes and linking them to reference documents and courses.

It manages:

  • Documents – official OBE‑related documents (vision, mission, curriculum, policies).
  • PEOs – Program Educational Objectives defined at program level.
  • PLOs – Program Learning Outcomes, mapped to PEOs.
  • CLOs – Course Learning Outcomes, mapped to PLOs and courses and later used in Custom Quest exams.

Together, these elements ensure that exams, courses, and programs are aligned with institutional outcomes and accreditation requirements.

5.2 Documents

Navigation: OBE System → Documents

Figure 21: Documents – list view

Purpose:

Maintain official reference documents that define and justify the outcome structure (vision documents, curriculum documents, OBE manuals, etc.).

List view columns:

  • Title – document title (e.g., “Vision Document”).
  • Reference / Number – internal reference or number (e.g., 11100).
  • Version – version number (e.g., 1.0).
  • Type – document type (e.g., Curriculum, Vision, Mission, Policy).
  • Effective Date – date from which this version is valid.
  • Status – Unpublished / Published.

Typical actions:

  • Create a new document:
    1. Click “Create”.
    2. Enter Title, Reference / Number, Version, Type, and Effective Date.
    3. Attach the actual file if your configuration supports it.
    4. Set Status to Draft/Unpublished until approved, then to Published.
    5. Save.
  • Link Documents as “Reference Document” in PEO, PLO, and CLO records to show which document defines each outcome.

5.3 PEOs (Program Educational Objectives)

Navigation: OBE System → PEOs

Figure 22: PEOs – list view

Purpose:

Define long‑term Program Educational Objectives for a degree program (what graduates are expected to achieve several years after graduation).

List view columns:

  • PEO – code or reference (e.g., PEO‑1).
  • PEO Title – short title of the objective.
  • PEO Description – detailed description (e.g., “Goes through the Project”).
  • Status – Unpublished / Published.
  • Reference Document – link to the Document record that describes this PEO.

Typical workflow:

  1. Academic quality team creates PEOs based on program vision and accreditation standards.
  2. Each PEO is linked to the relevant Document (e.g., Vision Document or Curriculum document).
  3. PEOs are set to Published once approved.
  4. PLOs are then mapped to these PEOs to show alignment.

5.4 PLOs (Program Learning Outcomes)

Navigation: OBE System → PLOs

Figure 23: PLOs – list view

Purpose:

Define observable and measurable Program Learning Outcomes – what students should be able to know or do at the time of graduation.

List view columns:

  • PLO Title – name of the outcome (e.g., “Graduate Attribute 2”).
  • Mapped PEO – PEO(s) that this PLO supports.
  • PLO Description – detailed description.
  • Status – Unpublished / Published.
  • Reference Document – document where this PLO is formally defined.

Typical usage:

  • For each PLO, specify which PEO(s) it supports using the Mapped PEO field.
  • Use status to control which PLOs are active for current accreditation cycles.
  • Ensure every PLO has a Reference Document for audit traceability.

5.5 CLOs (Course Learning Outcomes)

Navigation: OBE System → CLOs

Figure 24: CLOs – list view

Purpose:

Define Course Learning Outcomes for each course and link them to Program Learning Outcomes (PLOs).

List view columns:

  • CLO Title – identifier or code (e.g., “1”).
  • Description – short description of the outcome.
  • Taxonomy Level – cognitive level (e.g., Applying, Understanding, Analyzing).
  • Course – related course (e.g., “Database Management Systems”).
  • Associated PLO – PLO that this CLO contributes to (e.g., “Graduate Attribute 2”).
  • Status – Unpublished / Published.
  • Reference Document – the document where this CLO is described.

Typical workflow:

  1. For each course, define a set of CLOs that describe what students will learn.
  2. Assign a Taxonomy Level to reflect the expected cognitive level.
  3. Link each CLO to the appropriate PLO using “Associated PLO”.
  4. Link to the relevant Document (for example, course specification).
  5. Set Status to Published after academic approval.

Connection with Custom Quest:

In Exam Builder, questions are tagged with CLOs; this enables CLO Analysis and ensures that exams assess the intended course outcomes defined here.

5.6 OBE Data Flow and Usage

The OBE System module works in the following logical order:

  1. Documents
    • Store institutional and program‑level reference documents: vision, mission, curriculum, OBE policies, etc.
    • These documents are attached to outcomes as formal evidence.
  2. PEOs
    • Use these documents to define Program Educational Objectives for each program.
    • Approved PEOs are linked to one or more documents.
  3. PLOs
    • For each program, define PLOs and map them to the PEOs they support.
    • This shows how day‑to‑day learning outcomes contribute to long‑term objectives.
  4. CLOs
    • For each course, define CLOs and map each CLO to one PLO, with taxonomy levels.
    • CLOs are linked to course specifications and are later used in Custom Quest exams.
  5. Assessments (via Custom Quest)
    • Exam questions are linked to CLOs in Exam Builder.
    • CLO Analysis in Custom Quest reports how much of each CLO is assessed and with how many marks.
    • This closes the loop between institutional documents, program objectives, learning outcomes, and actual assessments.

Overall benefits:

  • Clear alignment between university vision, program objectives, learning outcomes, courses, and exams.
  • Strong evidence for accreditation bodies and quality assurance audits.
  • Ability to trace any exam question back to the CLO, PLO, PEO, and Document that justify it.
